# UMI与Diffusion Policy关键问题解析

本文档整理了关于UMI数据处理、Diffusion Policy数据要求、相对姿态原点以及建图与推理流程的核心讨论内容。

---

## 一、Diffusion Policy数据要求与格式

### 1.1 输入观察数据 (Policy Input)

| 数据类型 | 格式 | 说明 |
|---------|------|------|
| **RGB图像** | 224×224 或 224×224×2 | 单目(1×)或双目(2×)鱼眼图像，原始鱼眼图像不做畸变校正 |
| **相对EE姿态** | 6DoF (位置+旋转) | 相对于当前末端执行器姿态的历史轨迹 |
| **夹爪宽度** | 连续值 | 软体手指的当前开度 |

### 1.2 输出动作数据 (Action Output)

| 数据类型 | 格式 | 说明 |
|---------|------|------|
| **相对EE姿态序列** | 6DoF × T | 预测的未来T个时间步的相对末端执行器姿态 |
| **期望夹爪宽度** | 连续值 | 目标开度 |

### 1.3 关键超参数配置

来自论文TABLE A1（不同任务的配置）：

| 参数 | 准静态任务 | 动态任务 | 说明 |
|------|-----------|---------|------|
| **I-To** (图像视界) | 2 | 2 | 输入的历史图像帧数 |
| **P-To** (本体感知视界) | 2 | 2 | 输入的历史EE姿态帧数 |
| **Ta** (动作视界) | 6 | 6 | 预测的未来动作步数 |
| **Freq** (频率) | **10Hz** | **20Hz** | 准静态10Hz，动态任务需20Hz |
| **Speed** (速度) | 0.5x | 1.0x | 执行速度相对于演示速度 |
| **ImgRes** | 224×224 | 224×224 | 图像分辨率 |

### 1.4 Diffusion Policy训练配置

- **调度器**: DDIM (Denoising Diffusion Implicit Models)
- **训练扩散步数**: 50步
- **推理扩散步数**: 16步
- **视觉编码器**: ViT-B/16 或 ViT-L/14（使用CLIP预训练权重）
- **学习率**:
  - 动作扩散模型: 3e-4
  - 视觉编码器: 3e-5 (微调时降低10倍)

### 1.5 图像增强策略

```python
RandomCrop(scale=0.95)
RandomRotation(degrees=[-5.0, 5.0])
ColorJitter(brightness=±0.3, contrast=0.4, saturation=0.5, hue=0.08)
```

## 二、相对EE姿态的原点定义

### 2.1 核心定义

> **每个动作都是相对于"当前观察时刻的末端执行器姿态"（current EE pose at observation time）**

**原点**: 策略**最后一次观察**时的末端执行器姿态（p₀）

**动作序列**: [p₁-p₀, p₂-p₀, p₃-p₀, ...] 全部相对于同一个 p₀

```
时间线:
t0 (观察时刻) ── t1 ── t2 ── t3 ── t4 ── t5
│              ↓    ↓    ↓    ↓    ↓
│             Δp1  Δp2  Δp3  Δp4  Δp5
│              │    │    │    │    │
└──────────────┴────┴────┴────┴────┘
         全部相对于 p0 (t0时刻的EE姿态)
```

### 2.2 与Delta动作的对比

| 表示方式 | 定义 | 问题 |
|---------|------|------|
| **Delta** | 每个动作相对于前一个动作步骤 | 误差累积（drift）|
| **Relative (UMI使用)** | 每个动作相对于**同一个当前姿态** | 对跟踪错误鲁棒 |

## 三、训练与推理原点一致性保证

### 3.1 观察时刻对齐（PD1.1）

**训练时**：
- 从人类演示数据中提取
- 以**图像捕获时间戳**为基准
- 在该时间戳的**当前EE姿态**作为原点

**推理时**：
- 策略接收同步的观察序列
- 使用**最后一个观察步骤**的EE姿态作为原点
- 预测动作序列全部相对于这个原点

```
训练数据格式:
[Image_t0, EE_pose_t0, Gripper_width_t0] ──→ [Δpose_t1, Δpose_t2, ..., Δpose_t6]
                                ↑
                              原点

推理过程:
[Image_t0, EE_pose_t0, Gripper_width_t0] ──→ [Δpose_t1, Δpose_t2, ..., Δpose_t6]
                                ↑
                              原点
```

### 3.2 延迟补偿确保一致性（PD1.2）

**关键机制**：**提前发送命令** + **丢弃过时动作**

```
时间轴:
观察时刻 t0 ── 推理延迟 ── 输出动作 ── 执行延迟 ── 实际执行
     │              │            │            │
     └──────────────┘            ↓            ↓
          原点 p0         [过时动作丢弃]   [执行有效动作]
                              │
                              ↓
                        只执行时间戳正确的动作
```

**具体保证**：
1. 策略预测从最后一个观察步骤开始的动作序列
2. 前几个预测动作因延迟而过时，**直接丢弃**
3. 只执行时间戳与期望到达时间匹配的动作
4. 执行时使用**当前实际EE姿态**作为基准应用相对动作

### 3.3 相对轨迹作为本体感知（PD2.2）

当观察视界设为2时：
- 输入不仅包含当前图像
- 还包含**历史EE姿态**（表示为相对于当前原点的轨迹）
- 这实际上向策略提供了**速度信息**

```
观察输入 (视界=2):
- Image_t0
- Relative_EE_pose_t(-1)  [相对于t0的姿态]
- EE_pose_t0              [原点，值为0]
- Gripper_width_t0
```

## 四、建图与推理的关系

### 4.1 核心结论

| 阶段 | 是否需要预建地图 | 目的/影响 |
|------|-----------------|-----------|
| **训练数据采集** | ✅ **需要** | 提高SLAM跟踪质量，确保双手数据在同一坐标系 |
| **推理/部署** | ❌ **不需要** | 相对动作表示允许实时初始化 |
| **不建地图的影响** | 可能影响成功率 | 初始化更脆弱，跟踪失败风险增加 |

### 4.2 为什么训练时"必须"建图？

文档明确说明（PD2.3）：

> "对于每个新场景：首先收集**建图视频**构建场景地图，然后所有演示都在该地图上**重新定位**"

**核心原因**：
- **双手协调**：两个夹爪的视频分别重定位，需要**共享坐标系**才能计算夹爪间相对姿态
- **数据质量**：避免每次演示都重新初始化SLAM，减少跟踪失败导致的数据浪费

```
训练数据采集:
建图视频 ──→ 创建地图 ──→ 演示1 ──→ 在同一地图上重定位
                        ──→ 演示2 ──→ 在同一地图上重定位
                        ──→ 演示N
                           ↑
                    共享坐标系确保双手相对姿态准确
```

### 4.3 推理时"可以"不建图的原理

UMI的两项关键改进使实时初始化成为可能：

| 改进 | 作用 |
|------|------|
| **Map as Initialization** | 如果有地图，加载现有地图作为初始值 |
| **Marker-enhanced Initialization** | 建图时可用基准标记辅助（演示时移除）|

**推理时的灵活性**：
- 有地图 → 更鲁棒的初始化
- 无地图 → ORB-SLAM3尝试实时初始化（可能更慢、更脆弱）

### 4.4 实时推理流程（无需预建图）

```
实时推理循环:
┌─────────────────────────────────────────┐
│ 1. 相机捕获当前帧 + IMU数据              │
│ 2. ORB-SLAM3视觉-惯性优化                │
│    - 如果无地图: 初始化新地图             │
│    - 如果有地图: 在地图上重定位           │
│ 3. 输出当前6DoF EE姿态                   │
│ 4. 计算相对于当前姿态的目标动作           │
│ 5. 执行相对动作                          │
└─────────────────────────────────────────┘
```

### 4.5 成功率影响的具体场景

**可能降低成功率的情况**：
- **快速运动启动**：ORB-SLAM3初始化在快速运动下"尤其脆弱"
- **特征贫乏环境**：户外、白墙、重复纹理等
- **长时间运行**：累积漂移可能导致跟踪丢失

**Zero-shot泛化实验的暗示**：

> "policies learned via UMI **zero-shot generalize to novel environments**"

这说明系统**设计目标**就是能在无预建图的新环境运行，但"能运行"不等于"最佳性能"。

### 4.6 实用建议

| 场景 | 推荐做法 |
|------|---------|
| **固定场景（实验室/家庭）** | 预建地图 → 更高的跟踪鲁棒性 → 更高成功率 |
| **野外/新环境（Zero-shot）** | 无需建图 → 依赖实时初始化 → 可能成功率略低但仍可工作 |

## 五、校准无关性（Calibration-free）

### 5.1 核心优势

> "执行期间移动机器人基座不会影响任务表现（只要物体仍在可达范围内）"

**原因**：
- 动作是**相对**的，不依赖于全局坐标系
- 训练数据的原点和推理时的原点都是**相对于自身**的
- 无需精确的相机-机器人标定

```
场景: 机器人在执行过程中基座被移动

传统绝对动作:
全局坐标系 ──→ 基座移动后全局坐标改变 ──→ 动作失效

UMI相对动作:
当前EE姿态 ──→ 基座移动后重新定位 ──→ 动作仍然有效
    ↑
原点始终跟随实际EE位置
```

### 5.2 双手协调的特殊处理（PD2.3）

对于双手任务，UMI还使用**夹爪间相对姿态**：

- 两个夹爪在**共享地图**上分别重定位
- 计算两个夹爪之间的**相对姿态**
- 提供给策略作为额外的本体感知信息

这确保即使两个夹爪是独立跟踪的，它们之间的协调也是准确的。

## 六、总结

### 6.1 关键设计决策

| 方面 | 实现机制 |
|------|---------|
| **原点定义** | 最后一个观察步骤的当前EE姿态 |
| **训练-推理一致性** | 相同的数据格式：相对轨迹 |
| **延迟处理** | 提前发送命令 + 丢弃过时动作 |
| **鲁棒性保证** | 相对表示避免误差累积，不依赖全局坐标系 |

### 6.2 设计权衡

1. **训练阶段**：预建地图是"**强烈推荐的必需品**"（特别是双手任务）
2. **推理阶段**：预建地图是"**可选的优化**"，系统具备无地图运行的能力
3. **成功率影响**：无地图时初始化更脆弱，可能影响成功率，但UMI的相对动作设计确保即使跟踪有误差也不会累积
